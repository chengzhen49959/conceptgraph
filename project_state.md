## 1. Where We Are Now

**Fixed — worker OOM on large-PDF parse → text-layer extraction + LLM Markdown cleanup, 2026-06-30:**
- Root cause of the recurring **"worker offline"**: PDF parsing rendered pages through PyMuPDF4LLM's layout/table path, which holds heavy per-page structures in memory, so even a **single** ~75-page paper exceeded the 512MB Render worker → **OOM → SIGKILL** mid-parse. SIGKILL skips `except/finally`, so the doc froze at `parsing` and the arq health key lapsed (→ "worker offline" banner + `GET /api/health/worker` `{online:false}`); `on_startup` resume re-enqueued the same doc straight back into the same OOM — a crash-loop. (The earlier `parse_concurrency=1` semaphore bounded *concurrent* parses but cannot fix a lone big PDF that OOMs on its own — confirmed by live Render events still showing `oomKilled 512Mi` after it deployed. Upgrading the worker plan was explicitly ruled out by the user.)
- Fix (root cause): **stop rendering.** `parse._pdf_to_text` now reads the PDF **text layer** (`page.get_text()`) — no bitmap, no layout model — so parse memory is flat regardless of page count and the worker fits 512MB. The cheap extraction loses heading/table structure, so the worker then reformats it into clean Markdown with an LLM (`app/ai/cleanup.clean_markdown`, `parse_clean_model` = gpt-5.4-mini, PDFs only): the raw text is split into ~12k-char chunks on paragraph boundaries and each is reformatted in parallel **verbatim** (no summarising, so embeddings/citations stay faithful), with a **per-chunk raw fallback** (model failure / refusal / cleaned output `< 0.6×` its input → keep the raw chunk) so cleanup never loses content or blocks ingest. Kill switch `PARSE_LLM_CLEANUP=false` reverts to raw extraction with no per-doc model spend. `_PARSE_SEMAPHORE`/`parse_concurrency` are retained only as a gentle CPU bound (memory is no longer the limit). (`backend/app/services/parse.py`, `backend/app/ai/cleanup.py`, `backend/app/config.py`, `backend/app/worker.py`.)

**Done — ingestion pipeline (arq worker + Redis + S3), end to end:**
- parse (**PyMuPDF4LLM** → layout-aware Markdown for PDF, multi-column reading order **preserved by rendering each column under its own cropbox** (`_pdf_to_markdown` — the engine otherwise y-sorts a table-split two-column page's fragments and scrambles the reading order + merges side-by-side tables; columns detected via `column_boxes`, tiled gap-free) + structural headings, References section stripped, **then layout-engine noise stripped** — figure/formula `==> picture … intentionally omitted` placeholders + `----- picture text -----` OCR'd chart-axis garble, line-break hyphenation rejoined, bare page numbers + Unicode/blank-run normalised, all marker/line-anchored so no substantive text is touched; direct for MD/txt) → chunk (hash-idempotent) → embed chunks → extract concepts + relations (OpenAI structured output, per chunk, **pure recall** — every named concept, salience deferred to the gate) → **doc summary** → **core-concept gate** → **merge/dedup** → build edges (relation + **co-occurrence**) → **clustering**, with `documents.status` advancing **at every stage** (`pending → parsing → chunking → embedding → extracting → merging → clustering → done|failed`) so the frontend can show the live phase, not just a spinner.
- Core-concept gate (reduce-pass salience, `app/ai/select_core.py`): per-chunk extraction is blind to the whole doc, so related-work / bibliography passages each yield noise nodes. Centrality is global, so one document-level LLM call decides it. It judges each candidate against the document's **framing** — title + abstract + intro + conclusion, sliced positionally from the parsed text by `app/services/thesis.py` (fixed front/tail char windows, **not** section parsing — slicing never fails to return an anchor). The framing replaced the aggregate doc summary as the arbiter: the summary stitches every passage together and surfaces incidental machinery (optimizer, LayerNorm) as if it were the subject; the authors' framing names only the contribution. Each candidate carries `freq` (passage count, weak hint) + **`grounded`** (name/alias literally present in the framing — a strong, deterministic core prior, but English-only: concept names are normalised to English while the framing stays in source language, so for non-English docs grounding goes quiet and the model reads the framing directly — the prompt is told an absent flag means nothing); the prompt applies an **aboutness** test (is the doc *about* it, not does the doc *depend on* it) and leans **drop** when unsure (precision > recall). Drops are document-scoped (other docs' nodes untouched) and fail-open (refusal/empty/empty-framing → keep all, summary as fallback anchor). Only kept concepts are embedded/merged.
  - **Two recall levers landed (2026-06-26), precision held.** Re-scored on 6 LLM papers by independent per-paper graders vs each paper's known core: **mean precision 9.58 (≥ the prior 9.2), mean recall 9.33 (up from 7.8)**. (1) **Extraction → pure recall** (`extract.py`): dropped the per-chunk salience clause so a chunk surfaces *every* named concept; the recall ceiling was in extraction, not the gate (BERT's *bidirectionality* — in its title — and *fine-tuning* were never extracted, so the gate never saw them; both now kept, BERT recall 10/10). The kept (encyclopedia / not-a-field / not-a-person / not-a-sentence) tests still define what *is* a concept; only the centrality clause moved to the gate. Downstream cost unchanged — only CORE concepts are embedded/resolved/edged. (2) **Survey relaxation** (`select_core.py`): a survey's core is a whole field, so the contribution aboutness test collapsed it to its ~5 section titles. `_is_survey` (deterministic title/self-description regex, no LLM) appends an addendum **only** for survey/review/overview docs that widens the test to keep every named sub-area, dropping only incidentals — contribution-paper prompt left byte-identical (false positive is the only way it touches a normal paper, and only loosens, never tightens). LLM-survey: kept 7 → 47, now surfacing CoT / in-context learning / RLHF / RAG / scaling laws / emergent abilities / instruction tuning. (3) **Gate call timeout fix**: pure recall pushed the survey to ~3000 candidates, and the gate's single document-wide call legitimately reasons for tens of seconds — under the shared client's 45s per-request timeout (tuned for fast per-chunk extraction) it timed out and failed the doc. The gate call now overrides its timeout to 300s (one retry); measured 39–74s on the survey, huge margin under the 1800s job ceiling.
- Merge/dedup: exact-name **and alias** fast path (no LLM) → pgvector ANN recall → `match_concept` LLM judge → fold description + re-embed + alias. The whole merge/edge/cluster section runs under a per-workspace Redis lock (no duplicate nodes under concurrency); an `INSERT … ON CONFLICT(workspace_id, lower(name))` is the race backstop. The lock TTL is **tied to `_JOB_TIMEOUT`** (`worker.py`): a shorter TTL expired mid-merge under a multi-PDF batch (`Cannot release a lock that's no longer owned`), which also let a second job grab the lock mid-merge and break F4 — fixed by pinning both to the same constant. `max_tries=3` so one transient Aurora/DNS blip mid-write under batch load retries instead of permanently failing the doc.
- **Global dedup sweep** (`app/services/dedup_sweep.py`, arq job `dedup_sweep_workspace`): repair pass for the greedy incremental merge. Re-blocks every concept pair by exact cosine, re-runs the `match_concept` judge, and **merges two existing nodes** (re-points mentions/edges/aliases incl. `kind='cooccur'`, folds the description, deletes the loser; union-find for transitive A≡B≡C). Same merge lock; `dry_run` previews without writing; idempotent. Verified end-to-end (`test/dedup_sweep_smoke.py`: plants a split duplicate → sweep folds it, no mention/edge loss, second run is a no-op).
- **Co-occurrence edges** (`worker.py` Phase 2): concept pairs sharing a chunk get an undirected `kind='cooccur'` edge, weight = co-mention count accumulated across chunks/documents. Hidden from the graph view (`routers/graph.py` filters `kind='relation'`), used only to densify the cluster graph. On the attention.md smoke: 26 concepts → 40 relation + 128 co-occur edges (≈39% of the complete graph, weight avg 1.27 / max 4), lifting clustering from 3 leaves / 0 parents to 5 leaves / 2 parents.
- Doc summary: per-chunk summaries aggregated into one document summary, embedded, stored on `documents.summary` — **search only**; the core-concept gate now anchors on the document framing (`app/services/thesis.py`), not this summary (it falls back to the summary only when the framing is empty).
- Clustering: two-level Leiden over the weighted graph — relation **and** co-occurrence edges (igraph/leidenalg, networkx-Louvain fallback) → leaf `cluster_id` + `parent_id` hierarchy; orphan concepts (no edge) rescued by embedding-kNN; LLM labels each cluster; recomputed per workspace after each ingest.
- AI adapter layer (`app/ai/`): embeddings, extract, core-concept gate, merge judge, labels, summarize. OpenAI-only, backend-only.

- **Orphan-concept reconciliation (2026-06-26) — closes mid-pipeline TODO (4):** the dashboard showed **7 ghost nodes** (BERT / RoBERTa / CC-News … from a RoBERTa paper) with **0 mention / 0 edge / 0 cluster** and the sidebar said *0 documents / 0 topics* — a workspace-state contradiction, not a render bug. Root cause: `resolve_concept` commits each created node in **its own transaction** (Phase 1), separate from the mentions+edges write (Phase 2) and clustering (Phase 3); an ingest that dies between Phase 1 and Phase 2 (timeout-cancel / worker death / error) leaves *extracted* concepts no document mentions.
